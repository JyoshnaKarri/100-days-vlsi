    Day 6 – Latches & Flip-Flops

    Day 6 – Latches & Flip‑Flops: The Memory Cells of Digital Design

    Definition

    The term latch derives from the Old English læccan, meaning "to seize or hold" – reflecting its ability to capture and retain a single bit of data. Flip‑flop is an onomatopoeic term from early computing, mimicking the sound of a circuit toggling between two stable states.

    In digital electronics, a latch is a level‑sensitive bistable multivibrator: its output follows the input while an enable signal is active, and latches (holds) the last value when enable is deactivated. A flip‑flop is an edge‑triggered variant: it samples the input only on a clock edge (rising or falling), ignoring input changes at all other times. Flip‑flops are the fundamental building blocks of synchronous sequential circuits – registers, counters, and finite state machines (FSMs).

    2. Core Principles (4 Foundational Pillars)

    • Bistability: Cross‑coupled NAND or NOR gates create positive feedback. Two stable states (0 or 1) are maintained indefinitely without input. This is the physical basis of memory.
    • Level vs. Edge Sensitivity: Latches are transparent when enable = 1 (or 0). Flip‑flops are edge‑triggered – they sample input only on a clock transition. Edge triggering eliminates race conditions and enables synchronous design (all state changes aligned to a global clock).
    • Characteristic & Excitation Tables: The characteristic table defines the next state Q(t+1) given current state Q(t) and inputs. The excitation table is the inverse: given desired Q(t)→Q(t+1), it shows required inputs. These tables are essential for state machine synthesis and flip‑flop conversion (e.g., JK to D).
    • Timing Parameters: Setup time (data must be stable before clock edge), hold time (data stable after clock edge), propagation delay (clock‑to‑output). Violating these causes metastability – an unpredictable, non‑digital state.

    3. Mechanism (Step‑by‑Step)

    D Latch (Level‑Sensitive)

    • Enable = 1: Output Q follows D (transparent).
    • Enable = 0: Output Q holds the last value (latched).

    D Flip‑Flop (Edge‑Triggered)

    • Rising clock edge (↑): Q = D (sampled).
    • Otherwise: Q unchanged.

    JK Flip‑Flop

    • J=0, K=0: Hold (Q unchanged).
    • J=0, K=1: Reset (Q=0).
    • J=1, K=0: Set (Q=1).
    • J=1, K=1: Toggle (Q = ~Q).

    T Flip‑Flop

    • T=0: Hold.
    • T=1: Toggle.

    5. Critical Analysis

    • Pros: Simple, reliable, foundational for sequential logic. Flip‑flops enable synchronous design, eliminating race conditions.
    • Cons: Latches are transparent – if used in a clocked system, they can cause race conditions and unintended latches. Modern synchronous designs use edge‑triggered flip‑flops exclusively for registers.
    • Modern Applications: Registers, counters, frequency dividers, state machines, memory arrays (SRAM uses latches as bit cells).
    • Academic Debates: Latch vs. flip‑flop in ASIC design – some low‑power designs use latches, but mainstream digital design favors edge‑triggered flip‑flops.

    6. Real‑World Uses

    • Registers: Multiple D‑FFs sharing a common clock (e.g., 8‑bit register in CPU).
    • Counters: Ripple counters (T‑FFs), synchronous up/down counters.
    • Frequency Dividers: T‑FF toggles each clock cycle, dividing frequency by 2.
    • State Machines: Traffic light controllers, sequence detectors.
    • Memory Arrays: SRAM uses cross‑coupled inverters (similar to latches) as bit cells.

    7. Conclusion & Synthesis

    Latches and flip‑flops are the atomic memory units of every digital system. Without them, no state retention – no processors, no counters, no computers. Understanding their operation, timing, and application is essential for any VLSI or digital design engineer. Mastery of sequential circuits is a prerequisite for advanced topics like pipelines, caches, and microarchitectures.

    Day 7 – JK Flip‑Flop: The Workhorse of Sequential Logic

    Day 7 – JK Flip‑Flop: The Workhorse of Sequential Logic

    1. Why JK Flip‑Flop?

    The JK flip‑flop improves upon the SR flip‑flop by eliminating the invalid state (S=1,R=1) and introducing a toggle mode (J=1,K=1). It is the most versatile sequential element, used in counters, frequency dividers, and finite state machines.

    2. Characteristic Table & Equation

    The characteristic table defines the next state Q(t+1) based on current state Q(t) and inputs J,K.

    JKQ(t)Q(t+1)Operation
    0000Hold
    0011Hold
    0100Reset
    0110Reset
    1001Set
    1011Set
    1101Toggle
    1110Toggle

    Characteristic equation: Q(t+1) = J · Q'(t) + K' · Q(t)

    3. Excitation Table (Inverse)

    Used for state machine synthesis: given desired Q(t)→Q(t+1), find required J,K.

    Q(t) → Q(t+1)JK
    0 → 00X (don't care)
    0 → 11X
    1 → 0X1
    1 → 1X0

    4. Master‑Slave JK Flip‑Flop (Race Condition Solution)

    A simple JK latch (level‑sensitive) oscillates when J=K=1 and clock is high – called a race condition. The master‑slave configuration solves it:

    • Master stage samples J,K on the positive clock edge.
    • Slave stage outputs the master’s state on the negative clock edge.
    • Result: Only one toggle per clock cycle, no oscillation.
            Clock ──┬── Master ──┬── Slave ── Q
                │            │
                └── Inverter ─┘ (slave clock inverted)

    5. Toggle Mode & Frequency Division

    • When J=1, K=1, output toggles on every clock edge → frequency divider by 2.
    • Cascade n JK flip‑flops (each in toggle mode) → divide frequency by 2ⁿ.
    • Used in binary counters, digital clocks, and frequency synthesizers.

    6. Real‑World JK Flip‑Flop ICs

    • 74LS73 – Dual JK flip‑flop with clear (TTL logic, used in counters).
    • 74HC109 – Dual JK with preset and clear (high‑speed CMOS).
    • CD4027 – CMOS dual JK (low power, battery‑friendly).

    7. Applications of JK Flip‑Flop

    • Binary counters – ripple and synchronous counters (e.g., 4‑bit up counter).
    • Frequency dividers – divide clock frequency for digital watches, microcontrollers.
    • Shift registers – JK configured as D flip‑flop (J=K) for data storage.
    • Sequence detectors – state machines for pattern recognition (e.g., 101 sequence).
    • Traffic light controllers – FSM with JK flip‑flops for each state.
    • Random number generators – Linear Feedback Shift Registers (LFSR) using toggle mode.
    • Debounce circuits – JK latch to clean noisy switch inputs.
    • PWM generators – JK flip‑flops create variable duty cycle signals.

    8. Synthesis

    The JK flip‑flop is the most functionally complete sequential element. It can be configured as any other flip‑flop (D, T, SR). Master it, and you master counters, dividers, and finite state machines – the backbone of digital systems.

    Day 8 – Registers & Shift Registers

    Day 8 – Registers and Shift Registers

    Topic: Sequential Logic – Data Storage and Movement

    Single flip‑flops store 1 bit. To store multiple bits or move data, we combine flip‑flops into registers and shift registers. These are the building blocks of CPU registers, memory interfaces, and serial communication.

    1. Parallel Registers

    A register is a group of D flip‑flops sharing a common clock and common reset. All flip‑flops update simultaneously on the clock edge.

    4‑bit Register Structure

          D3   D2   D1   D0
      │    │    │    │
      ▼    ▼    ▼    ▼
    ┌──┐ ┌──┐ ┌──┐ ┌──┐
    │FF│ │FF│ │FF│ │FF│  (common clock and reset)
    └──┘ └──┘ └──┘ └──┘
      │    │    │    │
      ▼    ▼    ▼    ▼
      Q3   Q2   Q1   Q0

    Operation

    • On rising clock edge, each flip‑flop copies its D input to Q output.
    • All bits are stored in parallel – no shifting.

    Applications

    • CPU registers (accumulator, program counter, general‑purpose registers)
    • Memory address registers (MAR)
    • Memory data registers (MDR)
    • I/O port latches

    2. Shift Registers

    A shift register is a cascade of flip‑flops where the output of one is connected to the input of the next. Data moves (shifts) left or right on each clock pulse.

    Types of Shift Registers

    Type Name Description Typical Use
    SISO Serial In – Serial Out One input, one output. Data enters bit by bit, exits bit by bit after N clocks. Time delay, random number generators (LFSR)
    SIPO Serial In – Parallel Out One input, multiple outputs. After N clocks, all N bits are available in parallel. Receiving serial data (e.g., from SPI sensor)
    PISO Parallel In – Serial Out Multiple inputs, one output. Load in parallel, then shift out serially. Transmitting parallel data (e.g., to a serial display)
    PIPO Parallel In – Parallel Out Load and read in parallel. Behaves like a simple register. Data storage without shifting

    Day 9 – Synchronous Counters & Finite State Machines

    Day 9 – Synchronous Counters & Finite State Machines (FSM)

    | Topic: Sequential Logic – Counters and State Machines

    Synchronous counters eliminate ripple delay by clocking all flip‑flops simultaneously. Finite state machines (FSM) are the backbone of all sequential control – from traffic lights to CPU control units.

    1. Synchronous Counters

    All flip‑flops receive the same clock. Toggle conditions are derived from lower‑order bits using combinational logic.

    2‑bit synchronous up counter (using T flip‑flops)

    • T0 = 1 (always toggles)
    • T1 = Q0 (toggles only when Q0 = 1)
    Clock ──┬── T‑FF0 ── Q0 (LSB)
        │
        └── T‑FF1 (T1 = Q0) ── Q1 (MSB)

    4‑bit synchronous up counter toggle equations

    • T0 = 1
    • T1 = Q0
    • T2 = Q1 · Q0
    • T3 = Q2 · Q1 · Q0

    These equations ensure each bit toggles exactly when all lower bits are 1 – same as binary addition.

    Mod‑N counters

    Counts from 0 to N‑1 and resets. For a mod‑6 counter (0→1→2→3→4→5→0):

    • 3 flip‑flops (2³ = 8 ≥ 6)
    • Detect state 6 (binary 110) using a NAND gate on Q2 and Q1.
    • Connect NAND output to asynchronous clear or synchronous load.

    2. Ring Counter & Johnson Counter

    Ring counter

    A shift register with feedback from the last output to the first input. Only one '1' circulates.

    • N flip‑flops → N states.
    • Example (4‑bit): 1000 → 0100 → 0010 → 0001 → 1000...
    • Applications: State machines, timing generators, control sequencers.

    Johnson counter (twisted ring)

    Feedback from the inverted output of the last flip‑flop.

    • N flip‑flops → 2N states.
    • Example (3‑bit): 000 → 100 → 110 → 111 → 011 → 001 → 000...
    • Applications: Decoders, frequency dividers, sequence generators.

    3. Self‑Starting Counters

    A counter that reaches the valid sequence from any initial state (including unused states).

    • Prevents lock‑up in illegal states – critical in real hardware.
    • Achieved by designing next‑state logic so that all unused states eventually lead to a valid state.
    • Real‑world importance: Power‑up or glitch recovery – without self‑starting, the circuit may freeze.

    4. Finite State Machines (FSM)

    Two states are equivalent if they produce the same output and lead to equivalent next states for all input combinations. Reducing states simplifies the circuit.

    5. Real‑World Applications

    • Synchronous counters: Digital clocks, event counters, frequency meters, program counters in CPUs.
    • Ring/Johnson counters: Address decoding, LED chasers, walking‑one patterns, bus arbitration.
    • Self‑starting counters: Any safety‑critical system (automotive, medical) where lock‑up is unacceptable.
    • FSM (Mealy): Serial communication (UART, SPI), pattern detectors, encryption engines.
    • FSM (Moore): Traffic light controllers, vending machines, elevator control, microwave oven timers.
    • State reduction: Minimizing chip area in ASIC design, lower power in portable devices.

    Day 10 – Finite State Machines & Memory

    Day 10 – Finite State Machines (FSM) & Memory Basics

    Topic: Sequential logic applications, memory, and programmable devices.

    1. Finite State Machines (FSM) – Mealy vs Moore

    An FSM is a sequential circuit that transitions between a finite number of states based on inputs and current state. Two classic types:

    FeatureMealy FSMMoore FSM
    Output depends onPresent state + inputsPresent state only
    SpeedFaster (output changes without clock)Slower (output changes on clock edge)
    Number of statesGenerally fewerGenerally more
    ExampleUART receiver, sequence detectorTraffic light controller

    Advantages & Disadvantages:

    • Mealy: ✅ Fewer states, faster response. ❌ Output can have glitches if inputs change asynchronously.
    • Moore: ✅ Output stable, glitch‑free. ❌ May require more states, slower.

    2. FSM Design Examples

    🚦 Traffic Light Controller (Moore)

    • States: Red → Red‑Yellow → Green → Yellow → Red.
    • Outputs: Control signals to red, yellow, green lamps.
    • Timers: Counters generate delays (e.g., 30s red, 25s green, 5s yellow).
    • Advantage: Safe, outputs change only on clock edges – no glitches.
    • Disadvantage: More states than a Mealy equivalent (if one existed).

    💸 Vending Machine (Mealy)

    • Inputs: Coin (₹1, ₹2, ₹5), product select.
    • Outputs: Dispense product, return change.
    • States: 0, 5, 10, 15, 20 rupees.
    • Advantage: Fewer states, immediate output on coin insertion.
    • Disadvantage: Output can glitch if coin input changes during clock.

    🔍 1011 Sequence Detector (Mealy)

    • Purpose: Detect pattern "1011" in a serial bit stream (overlapping allowed).
    • States: S0 (none), S1 (1), S2 (10), S3 (101), S4 (1011).
    • Use: Frame detection in UART, start‑of‑packet in protocols.
    • Advantage: Minimal states (5).

    Topic: RTL Design with Verilog – Theory, Syntax, and Examples

    This page provides a complete reference for the core concepts of Verilog HDL. Each topic includes syntax, code snippets, ASCII diagrams, advantages/disadvantages, and real-world applications. Use this as a study guide before writing your own modules.

    1. Module and Port Declaration

    What it is: The basic building block. A module encapsulates logic with input/output ports.

    ┌─────────────────────────────────┐ │ module my_design ( │ │ input wire a, b, │ │ output reg y │ │ ); │ │ // logic inside │ │ endmodule │ └─────────────────────────────────┘ 
    module half_adder (
    input a, b,
    output sum, carry
);
    assign sum = a ^ b;
    assign carry = a & b;
endmodule
    • Advantages: Reusable, hierarchical, clear interface.
    • Disadvantages: Cannot have two modules with same name.
    • Use: Every Verilog design starts with a module.

    2. Data Types: wire vs reg

    • wire: Continuous assignment (combinational). Cannot store value inside always. Default value: high impedance (z).
    • reg: Can store value. Used inside always blocks. May be combinational or sequential.
    wire a;                // continuous
reg b;                 // can be used in always
assign a = x & y;      // wire
always @(*) b = x | y; // reg used combinationally
    • Advantages: wire is efficient for connections; reg gives flexibility.
    • Disadvantages: New users confuse reg with hardware register.
    • Use: wire for module interconnects; reg for sequential or procedural logic.

    3. Continuous Assignment (assign)

    Drives a wire with a combinational expression.

    assign #5 out = in1 & in2; // with 5 time unit delay
    • Advantages: Simple, always active.
    • Disadvantages: Cannot be used inside always blocks.
    • Use: Combinational logic like adders, muxes, decoders.

    4. Procedural Blocks: always

    Executes statements when an event occurs.

    // Sequential (flip-flop)
always @(posedge clk or posedge rst) begin
    if (rst) q <= 0;
    else q <= d;
end

// Combinational
always @(*) begin
    y = a & b;
end
    • Advantages: Models both combinational and sequential logic.
    • Disadvantages: Incorrect sensitivity lists cause simulation-synthesis mismatch.
    • Use: Flip‑flops, counters, FSMs.

    5. Blocking vs Non‑blocking Assignments

    TypeOperatorBehaviourUse
    Blocking=Executes in order (like C).Combinational logic inside always @(*)
    Non‑blocking<=All RHS evaluated before any LHS updated.Sequential logic (flip‑flops)
    always @(posedge clk) begin
    q1 <= d;      // non‑blocking
    q2 <= q1;     // old q1, not the new one
end
    ⚠️ Mixing blocking and non‑blocking incorrectly causes race conditions. Rule: use = for combinational, <= for sequential.

    6. if-else and case Statements

    // if-else (priority logic)

    if (sel == 2'b00) out = a;
    else if (sel == 2'b01) out = b;

always @(*) begin
    case (sel)
        2'b00: out = a;
        2'b01: out = b;
        2'b10: out = c;
        default: out = 0;
    endcase
end
    • Advantages: case is clearer for many conditions.
    • Disadvantages: if-else can create priority logic; case can create parallel mux.
    • Use: Multiplexers, decoders, ALU, FSM next‑state logic.

    Day 12 – Metastability & Clock Domain Crossing

    Day 12 – Metastability & Clock Domain Crossing (CDC)

    Topic: Reliable data transfer between clock domains.

    1. What is Metastability?

    A flip‑flop has two stable states: 0 and 1. If the data input changes too close to the clock edge (violating setup or hold time), the output may enter an unstable, unpredictable state – neither 0 nor 1, but an intermediate voltage that eventually settles to one of them after an unbounded delay. This is metastability.

    Clock ──┐ ┌─────┐ ┌─ │ │ │ │ Data ─┘ └───┬─┘ └─ (change too close to edge) │ Output ───────────┴─── metastable region ──── settles to 0 or 1

    Consequence: The downstream logic may see different values (some see 0, some see 1), causing system failure, data corruption, or crashes.

    2. When Does Metastability Happen?

    • Asynchronous inputs (e.g., button press, signal from another clock domain)
    • Violation of setup/hold time due to clock skew or data delay
    • Clock domain crossing (CDC) without proper synchronisation

    3. Solution: Two‑Flip‑Flop Synchroniser

    A two‑flip‑flop synchroniser (back‑to‑back flip‑flops) reduces the probability of metastability to an acceptable level (e.g., < 1e‑12 per year).

    Asynchronous input ──┬── FF1 ──┬── FF2 ── synchronised output │ │ clock ────────┘

    How it works:

    • First flip‑flop (FF1) samples the asynchronous input. It may become metastable.
    • Second flip‑flop (FF2) samples the output of FF1 after one clock cycle. By then, FF1 has almost certainly settled to a stable value.
    • FF2 output is safe to use in the destination clock domain.
    ⚠️ A two‑flip‑flop synchroniser does not prevent data loss. It only prevents metastable values from propagating.

    4. Clock Domain Crossing (CDC) – The Problem

    When data passes from one clock domain (CLK_A) to another (CLK_B) with different frequencies or phases, metastability is almost guaranteed unless proper CDC techniques are used.

    Common CDC scenarios:

    • Fast → slow clock (e.g., 200 MHz → 50 MHz)
    • Slow → fast clock (e.g., 25 MHz → 100 MHz)
    • Asynchronous (completely unrelated) clocks

    5. Basic CDC Techniques

    TechniqueUse caseAdvantageDisadvantage
    Two‑FF synchroniserSingle control signals (pulse, enable)Simple, low latencyCannot guarantee data loss for multi‑bit buses
    Handshake (request/ack)Slow control signals, multi‑bitReliable, losslessHigh latency, more logic
    FIFOMulti‑bit data streams (e.g., memory, UART)Lossless, high throughputLarger area, complex
    Gray code counterPointer crossing between domains (e.g., FIFO read/write pointers)Glitch‑free, safeLimited to incrementing values

    6. Why CDC Matters for VLSI

    • Most modern SoCs have multiple clock domains (CPU @ 2GHz, bus @ 500MHz, I2C @ 100kHz, USB @ 480MHz).
    • Improper CDC is the #1 cause of functional bugs in silicon – respins cost millions of dollars.
    • GATE EC/CS may not ask deeply, but interviews for VLSI roles always ask CDC basics.

    7. Example: Two‑FF Synchroniser Verilog Snippet (Conceptual)

    reg sync_ff1, sync_ff2;
always @(posedge clk_dest) begin
    sync_ff1 <= async_signal;
    sync_ff2 <= sync_ff1;
end
assign safe_signal = sync_ff2;

    The output safe_signal is synchronised and safe to use in the destination clock domain.

    8. Synchronizer Types – 2‑FF, 3‑FF, Pulse Synchronizer

    Different applications require different synchronizer structures.

    TypeStructureUse CaseLatencyLimitation
    2‑FF synchronizerTwo back‑to‑back flip‑flopsSingle control signals, level signals2 clock cyclesMay miss narrow pulses
    3‑FF synchronizerThree back‑to‑back flip‑flopsHigh‑reliability systems (automotive, medical)3 clock cyclesHigher latency, more area
    Pulse synchronizerToggle flip‑flop + XOR + 2‑FF九年VariableRequires minimum pulse width
    ✅ 2‑FF is sufficient for most applications. 3‑FF is used when MTBF requirements are extremely high (e.g., space, medical).

    9. Mean Time Between Failures (MTBF) – Why It Matters

    MTBF quantifies how often a metastable event propagates to cause system failure. The formula (simplified) is:

    MTBF = e^(t_metastable / τ) / (f_clk * f_data * W)
    • t_metastable : time allowed for metastability to resolve (usually one clock cycle)
    • τ : flip‑flop time constant (process‑dependent)
    • f_clk : sampling clock frequency
    • f_data : input data toggle frequency
    • W : timing window (setup + hold)

    Key takeaway: MTBF improves exponentially with faster flip‑flops (smaller τ) and slower clocks. A two‑FF synchroniser typically achieves MTBF > 1e‑12 years – safe for most designs.

    10. Gray Code for FIFO Pointers – Why Adjacent States Differ by 1 Bit

    When transferring multi‑bit pointers across clock domains (e.g., FIFO read/write pointers), multiple bits could change simultaneously, causing incorrect values if sampled during transition. Gray code solves this by ensuring only one bit changes between consecutive values.

    Binary: 0(000) → 1(001) → 2(010) → 3(011) → 4(100) (multiple bits change) Gray: 0(000) → 1(001) → 2(011) → 3(010) → 4(110) (only one bit changes)

    Application: FIFO read/write pointers are stored in Gray code before crossing clock domains. After synchronisation, they are converted back to binary for address generation.

    11. Handshake Protocol (Request/Acknowledge) – Reliable Transfer for Multi‑Bit Data

    When data must be transferred losslessly between clock domains, a handshake is used. It works as follows:

    1. Source domain asserts req (request) and places data on the bus.
    2. Destination domain samples req after synchronisation, captures the data, and asserts ack (acknowledge).
    3. Source domain sees synchronised ack and de‑asserts req.
    4. Destination domain sees req de‑asserted and de‑asserts ack.
    Source clk domain Destination clk domain req ──────────────────┬── synchroniser ──┬── captured data data ─────────────────┐ │ │ │ ack ──────────────────┴── synchroniser ──┘

    Advantage: Lossless, works for any data width. Disadvantage: High latency (multiple clock cycles).

    12. Summary of CDC Techniques (Quick Reference)

    ScenarioRecommended TechniqueKey Point

    Narrow pulse (fast to slow)Pulse synchroniserPreserves pulse width
    Multi‑bit pointer (FIFO)Gray code + 2‑FFOnly one bit changes at a time
    Lossless multi‑bit dataHandshake (req/ack) or FIFOReliable but higher latency
    Extreme reliability (auto, medical)3‑FF synchroniserHigher MTBF

    1. Parameterised Modules – Write Once, Use Any Width

    A parameter allows you to change the behaviour or bus width without rewriting the module. This is essential for building libraries.

    Example: 2:1 multiplexer that works for any bus width

    module mux2_param #(parameter WIDTH = 8) (
    input [WIDTH-1:0] a, b,
    input sel,
    output [WIDTH-1:0] y
);

    mux2_param #(.WIDTH(4)) mux4 (a4, b4, sel, y4);
mux2_param #(.WIDTH(16)) mux16 (a16, b16, sel, y16);

    Advantages: Reusable, less code, fewer errors. Synthesis tools flatten the parameter correctly.

    2. Generate Loops – Create Repetitive Logic Automatically

    The generate block with a for loop creates multiple instances or logic at compile time. It is perfect for regular structures like adders, decoders, or memory arrays.

    Example: 4‑bit ripple carry adder using generate (assumes a full_adder module exists)

    module rca4_generate #(parameter N=4) (
    input [N-1:0] a, b,
    input cin,
    output [N-1:0] sum,
    output cout
);
    wire [N:0] c;
    assign c[0] = cin;
    generate
        genvar i;
        for (i = 0; i < N; i = i + 1) begin : gen_adder
            full_adder fa (
                .a(a[i]), .b(b[i]), .cin(c[i]),
                .sum(sum[i]), .cout(c[i+1])
            );
        end
    endgenerate
    assign cout = c[N];
endmodule
    • genvar is a special variable only for generate loops.
    • The label gen_adder creates a named block (useful for hierarchy).
    • Loop bounds must be constant (synthesizable).

    3. Synchronous FIFO (First‑In‑First‑Out) – A Classic Design

    A FIFO is a buffer memory with separate read and write pointers. It is used for data buffering, clock domain crossing, and pipeline elasticity.

    Specifications: Depth = 8, width = 8 bits.

    • Inputs: clk, rst, wr_en, rd_en, din[7:0]
    • Outputs: dout[7:0], full, empty
    • Write when wr_en & !full → store at wr_ptr, increment pointer and count.
    • Read when rd_en & !empty → output from rd_ptr, increment pointer, decrement count.
    module fifo_sync #(parameter DEPTH=8, WIDTH=8) (
    input clk, rst,
    input wr_en, rd_en,
    input [WIDTH-1:0] din,
    output reg [WIDTH-1:0] dout,
    output full, empty
);
    reg [WIDTH-1:0] mem [0:DEPTH-1];
    reg [$clog2(DEPTH)-1:0] wr_ptr, rd_ptr;
    reg [$clog2(DEPTH):0] count; // extra bit for full/empty

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            wr_ptr <= 0;
            rd_ptr <= 0;
            count <= 0;
        end else begin
            if (wr_en && !full) begin
                mem[wr_ptr] <= din;
                wr_ptr <= wr_ptr + 1;
                count <= count + 1;
            end
            if (rd_en && !empty) begin
                dout <= mem[rd_ptr];
                rd_ptr <= rd_ptr + 1;
                count <= count - 1;
            end
        end
    end

    assign full  = (count == DEPTH);
    assign empty = (count == 0);
endmodule

    Testbench idea: Write 8 bytes, read them back, verify order. The FIFO is synthesizable and widely used in industry.

    1. Asynchronous FIFO (Gray‑Coded Pointers)

    Used to transfer data between different clock domains. The write and read clocks are independent. Pointers are converted to Gray code to avoid metastability.

    Day 19 – Logic Synthesis: From RTL to Gates (Complete Reference)

    Day 19 – Logic Synthesis: From RTL to Gates

    Complete reference – definitions, flow, constraints, synthesizable subset, and common pitfalls.

    1. What is Logic Synthesis?

    Logic synthesis is the process of converting a Register‑Transfer Level (RTL) description of a digital circuit (written in Verilog or VHDL) into a gate‑level netlist composed of standard cells (AND, OR, NOT, flip‑flops, etc.) from a technology library. It also performs optimization to meet area, speed, and power constraints.

    Why it matters: Without synthesis, RTL is only simulation. Synthesis produces the actual netlist that goes to placement and routing, and eventually to silicon.

    2. Inputs to Synthesis

    • RTL code – must be synthesizable (subset of Verilog).
    • Technology library – contains cell descriptions: function, area, delay (dependent on output load), power, and timing arcs.
    • Constraints – define design goals: clock frequency, input/output delays, false paths, multicycle paths. Usually written in SDC (Synopsys Design Constraints).

    3. Synthesis Flow – Three Main Steps

    RTL │ ▼ ┌─────────────┐ │ Translation │ → Generic Boolean network └─────────────┘ │ ▼ ┌─────────────┐ │ Optimization│ → Factoring, sharing, constant propagation, elimination └─────────────┘ │ ▼ ┌─────────────┐ │Technology │ → Map to library cells (NAND2, NOR2, DFF, etc.) │ Mapping │ └─────────────┘ │ ▼ Gate-level netlist
    • Translation: Parses RTL into an internal representation (e.g., AND/INVERTER graphs).
    • Logic optimization: Applies Boolean algebra laws to reduce area and delay. Example: !(a & b) → (~a | ~b) (DeMorgan).
    • Technology mapping: Replaces optimized generic gates with cells from the target library. It also chooses cell drive strengths.

    4. Constraints – The Language of Timing (SDC)

    Constraints tell the synthesizer how fast the design must run and the external environment.

    # Clock period = 10 ns → 100 MHz
create_clock -name clk -period 10.0 [get_ports clk]

# Input delay: external logic takes 3 ns before clk edge
set_input_delay -clock clk -max 3.0 [get_ports din*]

# Output delay: external logic needs 2.5 ns after clk edge
set_output_delay -clock clk -max 2.5 [get_ports dout*]

# False paths (no timing check, e.g., between unrelated clocks)
set_false_path -from [get_ports test_mode]

    Day 20 – Asynchronous FIFO (Gray‑coded pointers)

    Day 20 – Asynchronous FIFO (Gray‑coded pointers)

    Purpose: Safely transfer data between different clock domains.

    🔹 Working Principle

    • Problem: When multiple bits change at the same time across clock domains, metastability can occur.
    • Solution: Use Gray code for pointers – only one bit changes per count. Write and read pointers are converted to Gray, synchronised with two‑flip‑flop synchronisers, then compared to generate full/empty flags.
    • Dual‑port memory: Write on wclk, read on rclk.

    📊 Waveform Proof

    Write two bytes (0xA5, 0x5A) on fast clock (100 MHz), read back on slower clock (62.5 MHz). Gray pointer transitions are clean – no metastability.

    📁 Verilog Design (async_fifo.v)

    module async_fifo #(
    parameter DEPTH = 8,
    parameter WIDTH = 8
) (
    input wclk, wrst,
    input rclk, rrst,
    input wr_en, rd_en,
    input [WIDTH-1:0] din,
    output reg [WIDTH-1:0] dout,
    output full, empty
);
    localparam ADDR_WIDTH = $clog2(DEPTH);
    reg [WIDTH-1:0] mem [0:DEPTH-1];
    reg [ADDR_WIDTH:0] wr_ptr_bin, rd_ptr_bin;
    reg [ADDR_WIDTH:0] wr_ptr_gray, rd_ptr_gray;
    reg [ADDR_WIDTH:0] rd_ptr_sync1, rd_ptr_sync2, wr_ptr_sync1, wr_ptr_sync2;

    // Write pointer
    always @(posedge wclk or posedge wrst) begin
        if (wrst) {wr_ptr_bin, wr_ptr_gray} <= 0;
        else if (wr_en && !full) begin
            wr_ptr_bin <= wr_ptr_bin + 1;
            wr_ptr_gray <= (wr_ptr_bin + 1) ^ ((wr_ptr_bin + 1) >> 1);
        end
    end

    // Read pointer
    always @(posedge rclk or posedge rrst) begin
        if (rrst) {rd_ptr_bin, rd_ptr_gray} <= 0;
        else if (rd_en && !empty) begin
            rd_ptr_bin <= rd_ptr_bin + 1;
            rd_ptr_gray <= (rd_ptr_bin + 1) ^ ((rd_ptr_bin + 1) >> 1);
        end
    end

    // Synchronise read pointer to write clock domain
    always @(posedge wclk or posedge wrst) begin
        if (wrst) {rd_ptr_sync2, rd_ptr_sync1} <= 0;
        else {rd_ptr_sync2, rd_ptr_sync1} <= {rd_ptr_sync1, rd_ptr_gray};
    end
    wire [ADDR_WIDTH:0] rd_ptr_sync = rd_ptr_sync2;

    // Synchronise write pointer to read clock domain
    always @(posedge rclk or posedge rrst) begin
        if (rrst) {wr_ptr_sync2, wr_ptr_sync1} <= 0;
        else {wr_ptr_sync2, wr_ptr_sync1} <= {wr_ptr_sync1, wr_ptr_gray};
    end
    wire [ADDR_WIDTH:0] wr_ptr_sync = wr_ptr_sync2;

    // Memory write
    always @(posedge wclk)
        if (wr_en && !full) mem[wr_ptr_bin[ADDR_WIDTH-1:0]] <= din;

    // Memory read
    always @(posedge rclk)
        if (rd_en && !empty) dout <= mem[rd_ptr_bin[ADDR_WIDTH-1:0]];

    // Full flag (write Gray vs synchronised read Gray, MSBs differ)
    assign full = (wr_ptr_gray == {~rd_ptr_sync[ADDR_WIDTH:ADDR_WIDTH-1], rd_ptr_sync[ADDR_WIDTH-2:0]});
    // Empty flag (read Gray vs synchronised write Gray)
    assign empty = (rd_ptr_gray == wr_ptr_sync);

    Day 21 – Six Verilog Modules

    Day 21 – Verilog Modules: UART Tx/Rx, PWM, SPI, Debouncer, Binary to BCD

    Summary: Six synthesizable building blocks for digital design.

    1. UART Transmitter

    Working principle: Converts parallel byte to serial (start bit → 8 data bits LSB first → stop bit). Uses a counter derived from master clock to create correct baud rate (9600 from 50 MHz).

    module uart_tx #(CLK_FREQ=50000000, BAUD=9600) (
    input clk, rst,
    input [7:0] tx_data,
    input tx_start,
    output reg tx_busy,
    output reg tx
);
    localparam BIT_CNT_MAX = CLK_FREQ/BAUD;
    reg [12:0] bit_cnt;
    reg [2:0] bit_idx;
    reg [7:0] data_reg;
    reg [1:0] state;
    parameter IDLE=0, START=1, DATA=2, STOP=3;

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            state <= IDLE; tx <= 1; tx_busy <= 0;
        end else case (state)
            IDLE: begin
                tx_busy <= 0;
                if (tx_start) begin
                    data_reg <= tx_data;
                    tx_busy <= 1;
                    state <= START;
                    bit_cnt <= 0;
                end
            end
            START: begin
                tx <= 0;
                if (bit_cnt == BIT_CNT_MAX-1) begin
                    bit_cnt <= 0;
                    state <= DATA;
                    bit_idx <= 0;
                end else bit_cnt <= bit_cnt + 1;
            end
            DATA: begin
                tx <= data_reg[bit_idx];
                if (bit_cnt == BIT_CNT_MAX-1) begin
                    bit_cnt <= 0;
                    if (bit_idx == 7) state <= STOP;
                    else bit_idx <= bit_idx + 1;
                end else bit_cnt <= bit_cnt + 1;
            end
            STOP: begin
                tx <= 1;
                if (bit_cnt == BIT_CNT_MAX-1) begin
                    state <= IDLE;
                    tx_busy <= 0;
                end else bit_cnt <= bit_cnt + 1;
            end
        endcase
    end
endmodule

    Real‑world usage: Debug consoles, GPS modules, Bluetooth, serial bootloaders.

    2. UART Receiver

    Working principle: Oversamples at 16× baud rate, detects falling edge (start bit), samples each bit at the centre, assembles byte, outputs valid flag.

    module uart_rx #(CLK_FREQ=50000000, BAUD=9600) (
    input clk, rst, rx,
    output reg [7:0] data,
    output reg data_valid
);
    localparam BIT_CNT_MAX = CLK_FREQ/BAUD;
    reg [12:0] bit_cnt;
    reg [2:0] bit_idx;
    reg [7:0] shift;
    reg rx_meta, rx_sync;
    reg fall_edge;
    reg [1:0] state;
    parameter IDLE=0, START=1, DATA=2, STOP=3;

    always @(posedge clk) begin
        rx_meta <= rx;
        rx_sync <= rx_meta;
    end
    always @(posedge clk) fall_edge <= ~rx_sync & rx_meta;

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            state <= IDLE;
            data_valid <= 0;
            data <= 0;
        end else case (state)
            IDLE: begin
                data_valid <= 0;
                if (fall_edge) begin
                    state <= START;
                    bit_cnt <= BIT_CNT_MAX/2 - 1;
                end
            end
            START: begin
                if (bit_cnt == 0) begin
                    if (~rx_sync) begin
                        state <= DATA;
                        bit_idx <= 0;
                        bit_cnt <= BIT_CNT_MAX-1;
                    end else state <= IDLE;
                end else bit_cnt <= bit_cnt - 1;
            end
            DATA: begin
                if (bit_cnt == 0) begin
                    shift[bit_idx] <= rx_sync;
                    if (bit_idx == 7) begin
                        state <= STOP;
                        bit_cnt <= BIT_CNT_MAX-1;
                    end else begin
                        bit_idx <= bit_idx + 1;
                        bit_cnt <= BIT_CNT_MAX-1;
                    end
                end else bit_cnt <= bit_cnt - 1;
            end
            STOP: begin
                if (bit_cnt == 0) begin
                    if (rx_sync) begin
                        data <= shift;
                        data_valid <= 1;
                    end
                    state <= IDLE;
                end else bit_cnt <= bit_cnt - 1;
            end
        endcase
    end
endmodule

    Real‑world usage: Receiving data from GPS, serial sensors, console input.

    3. PWM Generator (8‑bit)

    Working principle: Free‑running counter, output = (counter < duty). Duty cycle = duty / 256.

    module pwm #(WIDTH=8) (
    input clk, rst,
    input [WIDTH-1:0] duty,

);

    Real‑world usage: LED dimming, motor speed control, digital‑to‑analog conversion.

    4. SPI Master (simple)

    Working principle: Transmits 8 bits over SPI (SCK, MOSI, CS). Uses a clock divider from master clock.

    module spi_master #(CLK_DIV=4) (
    input clk, rst,
    input start,
    input [7:0] tx_data,
    output reg busy,
    output reg sclk, mosi, cs
);
    reg [7:0] shift;
    reg [2:0] bit_cnt;
    reg [15:0] clk_div_cnt;
    reg sclk_en;
    reg [1:0] state;
    parameter IDLE=0, XFER=1, FINISH=2;

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            state <= IDLE; cs <= 1; sclk <= 1; busy <= 0;
        end else case (state)
            IDLE: begin
                cs <= 1; sclk <= 1;
                if (start) begin
                    shift <= tx_data;
                    bit_cnt <= 8;
                    cs <= 0;
                    busy <= 1;
                    clk_div_cnt <= 0;
                    sclk_en <= 1;
                    state <= XFER;
                end
            end
            XFER: begin
                if (sclk_en) begin
                    if (clk_div_cnt == CLK_DIV/2-1) begin
                        sclk <= ~sclk;
                        if (sclk == 0) begin
                            mosi <= shift[7];
                            shift <= {shift[6:0], 1'b0};
                            if (bit_cnt == 1) begin
                                state <= FINISH;
                                sclk_en <= 0;
                            end
                            bit_cnt <= bit_cnt - 1;
                        end
                        clk_div_cnt <= 0;
                    end else clk_div_cnt <= clk_div_cnt + 1;
                end
            end
            FINISH: begin
                cs <= 1;
                busy <= 0;
                state <= IDLE;
            end
        endcase
    end
endmodule

    Real‑world usage: Communicate with SPI sensors, flash memory, DACs, ADCs.

    5. Debouncer

    Working principle: Counts clock cycles while the input is stable. Output changes only after a fixed timeout, eliminating glitches from mechanical switches.

    module debounce #(TIMEOUT=10000) (
    input clk, rst,
    input noisy,
    output reg clean
);
    reg [$clog2(TIMEOUT)-1:0] cnt;
    reg stable;

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            cnt <= 0;
            clean <= 0;
            stable <= 0;
        end else begin
            if (noisy != stable) begin
                stable <= noisy;
                cnt <= 0;
            end else if (cnt < TIMEOUT-1) cnt <= cnt + 1;
            else clean <= stable;
        end
    end
endmodule

    Real‑world usage: Push‑button inputs, toggle switches, mechanical relay interfaces.

    6. Binary to BCD Converter (8‑bit)

    Working principle: Double‑dabble algorithm: shift bits left, if any BCD digit ≥5, add 3. After 8 shifts, result is three BCD digits (hundreds, tens, units).

    module bin2bcd (
    input [7:0] bin,
    output reg [11:0] bcd
);
    integer i;
    always @* begin
        bcd = 0;
        for (i = 0; i < 8; i = i + 1) begin
            if (bcd[3:0] >= 5) bcd[3:0] = bcd[3:0] + 3;
            if (bcd[7:4] >= 5) bcd[7:4] = bcd[7:4] + 3;
            if (bcd[11:8] >= 5) bcd[11:8] = bcd[11:8] + 3;
            bcd = {bcd[10:0], bin[7-i]};
        end
    end
endmodule

    Real‑world usage: Driving 7‑segment displays, digital clocks, any decimal output from binary data.

    module fifo_prog_almost #(
    parameter DEPTH = 8,
    parameter WIDTH = 8,
    parameter ALMOST_FULL_GAP = 1
) (
    input clk, rst,
    input wr_en, rd_en,
    input [WIDTH-1:0] din,
    output reg [WIDTH-1:0] dout,
    output full, empty,
    output almost_full
);
    localparam PTR_WIDTH = $clog2(DEPTH);
    reg [WIDTH-1:0] mem [0:DEPTH-1];
    reg [PTR_WIDTH-1:0] wr_ptr, rd_ptr;
    reg [PTR_WIDTH:0] count;

    always @(posedge clk or posedge rst) begin
        if (rst) begin
            wr_ptr <= 0; rd_ptr <= 0; count <= 0;
        end else begin
            if (wr_en && !full) begin
                mem[wr_ptr] <= din;
                wr_ptr <= wr_ptr + 1;
                count <= count + 1;
            end
            if (rd_en && !empty) begin
                dout <= mem[rd_ptr];
                rd_ptr <= rd_ptr + 1;
                count <= count - 1;
            end
        end
    end

    assign full = (count == DEPTH);
    assign empty = (count == 0);
    assign almost_full = (count >= DEPTH - ALMOST_FULL_GAP);
endmodule
